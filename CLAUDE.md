# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Panhandle: a shared shopping list + meal planner PWA for a household (solo, or up to 10 people â€” see Multi-tenant model), deployed entirely on Cloudflare (Pages + Worker + D1). Live at https://shopping.mohibb.com. Both the Pages project and the Worker track `main` directly via Cloudflare's native Git integration (not GitHub Actions) â€” pushing to `main` auto-deploys both.

The frontend is a Vite + React build (`src/`) â€” this is the live app. Cloudflare Pages builds it with `npm run build` â†’ `dist/`.

## Architecture

- **`worker/index.js`** â€” single-file Cloudflare Worker. Handles all `/api/*` routes, proxies everything else to the Pages project. Entire backend: routing, auth, and business logic in one `fetch` handler with sequential `if (path === ...)` checks.
- **`src/`, `package.json`, `vite.config.js`, root `app.html`** â€” the live frontend: three tabs (`src/tabs/ShoppingListTab.jsx`, `MealsTab.jsx`, `SettingsTab.jsx`). The Vite entry is named `app.html` (not `index.html`) so it builds to `dist/app.html` without colliding with `public/index.html`'s copy-through â€” see `docs/architecture-notes.md` for the full rationale and the pre-rewrite history. `scripts/compute-icon-offsets.mjs` is a manual dev tool for re-centering hand-drawn icons in `src/lib/itemIcons.js` â€” see its own header comment.
- **`public/`** â€” static assets Vite copies verbatim into `dist/`: `index.html` (unauthenticated marketing/landing page, links to `/app.html`), `changelog.html` (unauthenticated full-changelog page rendering `/CHANGELOG.md`, see Versioning), `manifest.json`, icons, `CHANGELOG.md`. `favicon.svg`/`favicon-32.png`/`favicon.ico` are rasterized from the same `src/design-system/assets/logo/panhandle-icon.svg` mark the app icons use (no ImageMagick/PIL in this environment, so built via a one-off Node script: Playwright Chromium to rasterize, a manual ICO writer to pack it â€” same Playwright-Chromium pattern as `scripts/compute-icon-offsets.mjs`) and linked from `app.html`/`index.html`/`changelog.html`'s `<head>`; regenerate by hand if `panhandle-icon.svg` ever changes.
- **`migrations/`** â€” SQL applied via Wrangler's D1 migrations runner, tracked in the `d1_migrations` table (see `wrangler.toml`). `0001_init.sql` is the full consolidated base schema; every later file is additive. New schema/data changes are just a new numbered `.sql` file â€” see `docs/architecture-notes.md` for the per-migration history, and "Applying migrations" below for how to actually run one.

### Deployment (Cloudflare Git integration)

- Both projects track `main` in the Cloudflare dashboard â€” no `.github/workflows/` deploy anything.
- Worker deploy command: `npx wrangler deploy`, reading `wrangler.toml`.
- Worker runtime secrets live in the Worker's dashboard (never in `wrangler.toml` or the repo): `JWT_SECRET`, `RESEND_API_KEY` (Resend email), `TURNSTILE_SECRET_KEY` (CAPTCHA on public endpoints), `FEEDBACK_EMAIL` (also reused as the VAPID contact, see Push notifications), `SUPERADMIN_USERNAMES` (comma-separated allowlist, see Multi-tenant model), `VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY` (Web Push signing keypair, see Push notifications).
- `wrangler.toml`'s `[triggers]` cron (`*/15 * * * *`, driving `runReminderPass` â€” see Push notifications) deploys along with everything else on push to `main`; no separate step is needed.

### Auth model (worker/index.js)

- Passwords: PBKDF2 (100k iterations, SHA-256) via Web Crypto (`hashPassword`/`verifyPassword`). Tokens: hand-rolled HS256 JWT (`signJwt`/`verifyJwt`), signed with `env.JWT_SECRET`.
- Every JWT carries `tv` (token_version); `requireAuth` checks it against the DB's current value, so a password change or admin/owner flag change invalidates other devices' tokens immediately even though those tokens haven't expired.
- Sliding expiry: authenticated responses include a fresh token in `X-Refresh-Token` (`mintToken`); `src/lib/api.js` re-stores it via `AuthContext`. `/change-password` and `/change-email` instead return the new token in the response body, since both bump `token_version` or rename the JWT's `sub`.
- No admin-bootstrap backdoor: `POST /register` (Turnstile-gated), `POST /auth/google` (matched on `google_sub`), and `POST /forgot-password`/`POST /reset-password` (single-use, hashed tokens, emailed via Resend) are the only ways to create or recover an account. `/register`, `/forgot-password`, `POST /feedback` are IP-rate-limited via the shared `rate_limit_attempts` table (`checkRateLimit`/`recordAttempt`).
- Three editable identity fields: `username` (always equals `email`, server-enforced), `email`, `name` (free-typed display name). `POST /change-email` renames the username everywhere it's copied by value (`list_items.added_by`, `meal_plan.responsible`, `recurring_schedule.responsible`, `users.created_by`, `password_resets.username`, `push_subscriptions.username` â€” see `renameUsername`), not a single-column update.
- `isSuperAdmin` (checks `env.SUPERADMIN_USERNAMES`) is orthogonal to and stronger than per-list `is_admin`/`is_owner`; gates `GET /admin/metrics` and `DELETE /admin/users/{u}`. Since that status comes solely from being named in the env var rather than a DB flag, both `DELETE /admin/users/{u}` and `DELETE /account` unconditionally refuse to delete *any* superadmin's account â€” no count check, no override â€” superadmins can't delete each other or self-delete, since the only way to actually remove one is a developer editing that env var by hand.
- Every user can `GET`/`DELETE /account` (self-delete, current-password + IP-throttle). The sole/last owner of a list self-deleting cascade-deletes the whole list instead of being refused (unlike the admin-facing removal endpoints, which refuse on the last owner) â€” except a superadmin, who can never self-delete at all (see above), regardless of ownership.

### Multi-tenant model

- Every user belongs to exactly one `list_id`; all shopping/meal/catalogue data is scoped by it server-side â€” `list_id`/`is_admin`/`is_owner` are **never** accepted as request input.
- `is_admin`/`is_owner` are independent 0/1 flags (a list can have multiple owners; "member" = neither). The JWT carries them for client display only â€” `requireAuth` re-reads from the DB every request. Any flag/`list_id` change bumps the target's `token_version`.
- Admin endpoints (`is_admin`): `GET /admin/users`, `POST /admin/users/{u}/reset-password`, `PATCH /admin/users/{u}/flags` (refuses to demote a list's last admin or remove a list's only owner â€” both counted per-list). These are scoped to the caller's own `list_id` â€” an admin can only see/reset/flag users in their own household â€” unless the caller is also superadmin, who acts across every list; a non-superadmin admin additionally can never reset the password or change the flags of a superadmin account, even one sharing their list (closes the cross-tenant escalation TODO #90 flagged). Owner endpoints (`is_owner`): `POST /list-users` (10-user cap, member flags server-forced to 0), `DELETE /list-users/{u}`. `GET /list-users` is open to any list member (drives the meal-responsible dropdown). Superadmin-only: `GET /admin/metrics`, `DELETE /admin/users/{u}`, `POST /admin/owners` (minting a brand-new household/owner account is an app-operator action, not "manage my list").
- New lists are seeded with the hardcoded `COMMON_ITEMS` array (710 common Norwegian items) at creation â€” a per-list copy, not a shared table (see "Catalogue sync" below for how existing lists stay in sync with later additions to the array).

### Data flow

- Shopping items reference a shared `item_catalogue` (name + category) so a name typed once is remembered with its category; `list_items` tracks bought/added-by state against a catalogue entry. `item_catalogue.times_bought`/`first_bought`/`last_bought` are durable, never-pruned counters powering `GET /catalogue/suggestions`. `DELETE /list/bought` (TODO #100, the "TĂ¸m handlede" end-of-trip sweep) deletes every bought `list_items` row for the list in one shot; it only touches `list_items`, so the catalogue rows and their durable stats survive and re-adding/suggestions are unaffected.
- Meals follow the same pattern: `meal_catalogue` holds known meal names, `meal_plan` assigns one meal + a responsible person per `plan_date` (upserted via `ON CONFLICT(plan_date)`). The frontend only ever navigates `[WEEK_MIN, WEEK_MAX]` = `[-1, 4]` weeks of the current week (`src/lib/mealUtils.js`); `GET /plan` opportunistically deletes rows older than 14 days on every read â€” this cleanup still piggybacks on normal traffic rather than the Worker's cron trigger (see Push notifications), which runs a separate, unrelated check. `meal_catalogue.times_planned`/`last_planned` power `GET /meals/suggestions`. `meal_plan.meal_id` is nullable with `ON DELETE SET NULL`, so deleting a catalogue meal unassigns it from a planned day (keeping `responsible`) instead of deleting the row. `meal_catalogue.labels` is a free-form, user-typed JSON string array, sanitized via `sanitizeLabels` (trimmed, capitalized, deduped case-insensitively).
- `recurring_schedule` holds at most one row per `list_id`+`day_of_week`, each with a `responsible` person â€” a standing weekly default, separate from `meal_plan`'s per-date assignments.
- Frontend polls `/list` and `/plan` every 7 seconds while the relevant tab is active (no websockets/push).
- Offline write durability (TODO #113): shopping-list writes are optimistic-UI-first, and a mutation that fails with a *network* error (not a server rejection) is persisted to a localStorage FIFO â€” `src/lib/writeQueue.js` (`ph_write_queue_v1`) â€” and replayed in order on reconnect, so an add/toggle/mark-important made mid-shop with no signal isn't lost. Scope is deliberately the three "while shopping" writes (add `POST /list`, toggle `POST /list/:id/toggle`, important `PATCH /list/:id`); edit/delete-from-catalogue stay online-only. An offline add gets a client temp id (`newTempId`, `tmp_` prefix) for its optimistic row; `flushQueue` replays FIFO and maps that temp id to the real one from the add's response (`POST /list` returns the new/merged row `id` for exactly this) so a toggle/important queued against a not-yet-synced item resolves. `ShoppingListTab.jsx`'s `loadList` flushes the queue *before* fetching and only overwrites local state with the server snapshot once the queue is drained (else it keeps the optimistic state); it also flushes on the `online` event, persists the optimistic list to `ITEMS_CACHE_KEY` on every change (survives a reload while offline), and shows a "usendte" count pill. Delivery is at-least-once (the server's identical-unbought-item merge absorbs a double-replayed add as a qty bump); the queue is cleared on logout (`AuthContext`) alongside `clearCache`. Unit-tested in `src/lib/writeQueue.test.js`.

### Push notifications (TODO #7 phases 1-2)

- Phase 1: Web Push subscribe/unsubscribe infrastructure plus a cron-driven "no meal planned for tomorrow" reminder. Phase 2: a weekly meal-plan reminder and an on-demand "get the other person's attention" ping. Batched item-added notifications remain a later, not-yet-built phase of the same TODO item.
- `migrations/0012_push_notifications.sql` adds `push_subscriptions` (keyed by `endpoint`, not `username`, since a subscription belongs to a browser/device, not an account â€” on a shared household device, whoever last (re)subscribed owns it; `POST /push/subscribe` upserts `ON CONFLICT(endpoint)`), `notification_settings` (one row per list, no row seeded â€” `GET /notification-settings` falls back to app-level defaults until a list's first `POST`), and `notification_log` (`UNIQUE(list_id, type, target_date)`, a dedup guard so the cron never sends the same list/day's reminder twice). `migrations/0014_notification_phase2.sql` adds `notification_settings.weekly_reminder_enabled`/`weekly_reminder_time` (`HH:mm`, same 15-minute-increment convention as `meal_reminder_time`; the weekly reminder's day is hardcoded to Sunday in code, not a column) and `notification_state` (`list_id, type, last_notified_at`, internal cron/endpoint bookkeeping â€” watermarks and cooldowns â€” kept separate from `notification_settings`'s user-facing preferences, mirroring `notification_log` already being internal-only).
- Endpoints (all behind the auth boundary â€” a user must already be logged in to reach the Settings/FAB controls, so there's no new public route): `GET /push/vapid-public-key`, `POST`/`DELETE /push/subscribe`, `GET`/`POST /notification-settings` (same any-list-member permission level as `/recurring`, not owner-gated, covers both reminder types), `POST /push/ping` (phase 2 â€” any list member, fixed message, fans out to every other subscribed device on the caller's list via `sendPushToList`'s `excludeUsernames`, rate-limited to once per 2 minutes per list via `notification_state`). A departing/removed member's `push_subscriptions` row is cleaned up in `DELETE /account`, `DELETE /admin/users/{u}`, and `DELETE /list-users/{u}` alongside the existing per-list cascades, since it's a live credential, not historical data; `notification_settings`/`notification_state` are cleaned up in the two list-deletion cascades (`DELETE /account`'s and `DELETE /admin/users/{u}`'s last-owner paths).
- `wrangler.toml`'s `[triggers]` cron (`*/15 * * * *`) drives `worker/index.js`'s exported `scheduled(event, env, ctx)`, which fires `runNotificationPass` and `checkCatalogueSync` (see "Catalogue sync" below â€” an unrelated concern that happens to reuse the same cron) as two separate `ctx.waitUntil(...)` tasks. `runNotificationPass(env, nowMs)` is a dispatcher calling two independently testable checks â€” `checkMealReminders` (phase 1) and `checkWeeklyReminders` (phase 2, gated to Sunday evenings only, via `osloLocalDateParts`'s `dow` field, 0=Mon..6=Sun matching `recurring_schedule.day_of_week`'s convention; fires only if the upcoming Mon-Sun week has *zero* planned meals â€” deliberately "completely unplanned," not "few," since a threshold would need arbitrary tuning and risks nagging a household that's already started) â€” both taking `now` as a parameter (rather than reading `Date.now()` internally) specifically so `tests/push-notifications.test.mjs` can exercise their dedup/skip/fan-out logic deterministically against a fake `env.DB` stub, without depending on wrangler dev's untested `--test-scheduled` scheduled-event simulation. `osloLocalDateParts`/`isReminderDue`/`addDaysIso` (also exported, unit-tested in `tests/worker-unit.test.mjs`) compute the current Europe/Oslo local time/day-of-week and calendar-date math via `Intl.DateTimeFormat`, which correctly follows Oslo's CET/CEST DST transitions; two known, accepted edge cases (documented in code, not fixed): a reminder time inside the skipped spring-forward hour silently doesn't fire that one day/year, and the repeated fall-back hour would double-fire if `notification_log`'s dedup constraint didn't absorb it as a no-op. `sendPushToList(env, list_id, payload, { excludeUsernames })` (exported, unit-tested) is the shared fan-out helper both checks and `POST /push/ping` use, factored out of phase 1's original inlined loop.
- Payload encryption uses `@pushforge/builder`, a small dependency-free (pure Web Crypto, no Node crypto shims) Web Push library â€” a deliberate, documented exception to this app's otherwise zero-external-crypto-deps convention (PBKDF2/JWT/RSA verification are all hand-rolled â€” see Auth model), since implementing RFC 8291's `aes128gcm` payload encryption by hand carries more security risk than a focused library does. The VAPID JWT's `sub` claim reuses the existing `FEEDBACK_EMAIL` secret rather than adding a new one.
- Frontend: `src/context/PushContext.jsx` holds subscribe/unsubscribe state and re-syncs the browser's current subscription to the server on every mount/token change (also how a subscription silently rotated while no tab was open gets its new endpoint picked up â€” see below). `src/components/settings/subpages/VarslerSubpage.jsx` (Settings â†’ "Varsler", one of the subpages in `SettingsRoot.jsx`'s two-tier nav) has the enable-notifications toggle and both reminder types' enabled/time controls. `src/tabs/ShoppingListTab.jsx`'s `FabMenu` gained a "Varsle husstanden" action calling `POST /push/ping` (a 429 from the per-list cooldown surfaces as a toast, since there's no cheap client-side way to know the cooldown is active). `public/sw.js` gained `push` (shows the notification), `notificationclick` (focuses/opens the app), and `pushsubscriptionchange` (browsers can silently rotate/invalidate a subscription outside of any 404/410 the server would see) listeners â€” the last of these only keeps the *browser-level* registration alive, since a service worker has no access to this app's auth token (it lives in `localStorage`, unreachable from a SW) to re-authenticate a server-side sync itself; that sync happens next time the app opens, via `PushContext`'s refresh.
- Web Push on iOS only works for the PWA added to the Home Screen (standalone display mode), not an ordinary Safari tab, and only iOS 16.4+ â€” `VarslerSubpage` shows a hint when it detects that case.

### Catalogue sync

- Rolling out a new common item is just editing the `COMMON_ITEMS` array (`worker/index.js`) and deploying â€” no separate migration file needed. `migrations/0018_catalogue_sync_state.sql` adds `catalogue_sync_state` (single row, `id=1`), which stores a hash of the last-synced `COMMON_ITEMS`. `checkCatalogueSync(env, items = COMMON_ITEMS)`, driven by the same 15-minute cron as the notification checks (see above â€” a separate `ctx.waitUntil` task in `scheduled()`, unrelated to notifications), hashes the current array on every tick and compares it to that stored hash: unchanged means an immediate no-op (one row read, no writes), changed means it re-runs an `INSERT ... ON CONFLICT(list_id, name) DO UPDATE SET category = excluded.category` backfill of every item into every existing list's `item_catalogue` (same non-destructive upsert shape as the old `migrations/0002_seed_catalogue.sql`/`0003_expand_catalogue.sql`, which this replaces), then updates the stored hash. A household's own custom (non-`COMMON_ITEMS`) catalogue entries are never touched. `items` is a parameter (not a hardcoded read of `COMMON_ITEMS`) specifically so `tests/catalogue-sync.test.mjs` can exercise it against a small fixture array via a fake `env.DB` stub, same pattern as the notification checks.

### Categories and people

`CATEGORIES` lives in `shared/categories.js`, imported by both `worker/index.js` and `src/lib/shoppingUtils.js` â€” same single-source pattern as `VERSION` (see Versioning). People are derived from `GET /list-users` (`src/context/ListUsersContext.jsx`), not hardcoded. `COMMON_ITEMS` (new-list catalogue seed) lives only in `worker/index.js`.

`CATEGORIES` is the *default* aisle order; a household can override it (TODO #105 â€” "match my store's layout"). The custom order is a shared per-list setting stored in the `category_order` table (`migrations/0017_category_order.sql`, one row per `list_id`+`category` with a `position`), read/written via `GET`/`POST /category-order` (any-list-member permission, same level as `/recurring` â€” it's the household's store layout, not a per-device preference). Both endpoints and the frontend run the stored order through `normalizeCategoryOrder` (also in `shared/categories.js`, unit-tested in `tests/worker-unit.test.mjs`), which merges any stored positions with any not-yet-placed categories in canonical order and drops unknown/duplicate names â€” so a partial or stale order (e.g. after `CATEGORIES` gains an entry) always resolves to a complete, valid permutation. Frontend: `src/context/CategoryOrderContext.jsx` loads it once on login and after a save (no polling â€” it changes rarely), falling back to `CATEGORIES` until loaded / on fetch failure; `ShoppingListTab.jsx` groups unbought items by this order instead of the constant, and `src/components/settings/subpages/ButikkSubpage.jsx` (Settings â†’ "Butikkoppsett") is the reorder editor: drag-to-reorder via framer-motion's `Reorder.Group`/`Reorder.Item` (already a dependency elsewhere only for swipe gestures â€” no new dependency added), with the original up/down buttons kept alongside since `Reorder` alone isn't keyboard-operable. `category_order` is cleaned up in the two list-deletion cascades (`DELETE /account`'s and `DELETE /admin/users/{u}`'s last-owner paths), like the other per-list tables.

### Versioning

Single version number, `VERSION` in `shared/version.js`, imported by both `worker/index.js` and `src/lib/version.js`. `GET /api/version` (public) exposes both app and API versions, so a mismatch on the Profile page means one half of a deploy is still in flight. `CHANGELOG.md` is auto-copied to `public/CHANGELOG.md` on every build (`scripts/sync-changelog.mjs`, wired into `prebuild`) â€” editing the root file is enough. `public/changelog.html` and `ChangelogModal` both render it (full history vs. the last few versions' compact titles, in a larger font, respectively â€” `ChangelogModal` deliberately doesn't dump the whole history, so it reads as a quick "what's new" rather than an archive) â€” see `docs/architecture-notes.md` for why they use separate parsers.

Version bump convention (`MAJOR.MINOR.PATCH`): bump MINOR for a release adding a new user-facing capability or completing a planned phase/epic; PATCH for fixes/tweaks/small additions; MAJOR only for breaking changes. `useDeployVersionCheck` (`src/hooks/useDeployVersionCheck.js`) compares a returning browser's last-seen `VERSION` against the current one and reacts differently depending on which part changed (`isFeatureVersionBump`, `src/lib/version.js`): a MAJOR/MINOR bump auto-opens `ChangelogModal` immediately, since that's an actual new capability worth surfacing; a PATCH-only bump just shows a dismissible toast with a "Hva er nytt?" button into the same modal, since a fix/tweak isn't worth interrupting for.

## Workflow conventions

When the user says "finish up" (or similar) on a branch with work ready to ship: sync with `main`, push, open a PR, wait for checks/review, then merge â€” without needing to ask at each step. CI (`.github/workflows/ci.yml`: lint, secret-scan, migration-numbering check, frontend build+unit-tests, backend unit-tests) doesn't gate deploys â€” Cloudflare's Git integration deploys independently of GH Actions results. So "waiting" means watching CI status, the Cloudflare deploy-preview bot comment, and review feedback before merging.

**What "wait for checks/review" actually means here.** There is no second human reviewer â€” this is a solo dev plus Claude â€” so "review" is **not** a human approval gate to wait on (waiting on one indefinitely, or silently skipping it, are both wrong). The real pre-merge signals are: CI status (below), the Cloudflare deploy-preview bot comment plus a hands-on click-through of that preview URL, and â€” only when the user explicitly asks for it â€” a Copilot review. Absent an explicit Copilot request, the deploy-preview click-through is the review.

**Never merge on red CI.** Because Cloudflare's Git integration deploys on push independently of GH Actions, a merge with failing lint/unit-tests still redeploys prod â€” only a Cloudflare *build* failure actually blocks a deploy, and nothing enforces the "watch CI" step. So treat red CI as a hard stop: don't merge until lint, secret-scan, migration-numbering, and the unit-test jobs are green (and, on the paths that trigger it, the integration job â€” see Testing conventions). A red run means "fix it or wait for it," never "merge anyway; it deploys regardless."

**Every PR description starts with a one-sentence, plain-English summary for a non-developer** â€” what changed and why it matters to someone using the app, no jargon (endpoint names, file paths, flag names). Put it first, above the technical `## Summary`/`## Test plan` detail.

**A PR that changes the app served to users doesn't merge without a version bump.** Every merge to `main` immediately redeploys the live app. The test is whether the change alters the **built output** â€” the Pages `dist/` bundle or the deployed Worker: frontend (`src/`, `public/`, `app.html`), the Worker (`worker/`), migrations, shared code both import (`shared/`), **and the build inputs that feed that output** â€” dependency changes (`package.json`/`package-lock.json`), `vite.config.js`, and any build-time script (e.g. `scripts/sync-changelog.mjs`, which runs in `prebuild`). For any of these, bumping `VERSION` and adding a matching `CHANGELOG.md` entry is part of what "ready to merge" means, not optional cleanup: do it in the same PR before merging.

**Internal-housekeeping-only changes do NOT get a version bump and are not surfaced to users.** If a PR leaves the built output byte-for-byte identical â€” `TODO.md`/`Todo_done.md`, `docs/`, this `CLAUDE.md`, `README.md`, tests (`tests/`, `src/**/*.test.*`), CI/workflow config (`.github/`), dev/preview-only config that never changes the production bundle or Worker, or manual dev-only tools that never run in the build (`scripts/compute-icon-offsets.mjs`, the one-off favicon script) â€” then the running app is identical after the merge, so there's nothing to "update" for users: leave `VERSION` and `CHANGELOG.md` alone. `CHANGELOG.md` is a user-facing "what's new" (copied into `public/` and rendered in-app), so an entry about internal notes is noise the user never asked to see. The test is strictly *does the built output change?* â€” not how big the diff is. (A change that touches both app output and docs is an app change: bump it.)

If changes related to anything in CLAUDE.md are made, the change should also be reflected in this file.

### Testing conventions

New pure-logic functions (worker helpers, `src/lib/*`) and new auth/permission-sensitive endpoints should get a unit or integration test added in the same PR â€” written after the behavior is working and validated on a deploy preview, not TDD-style. UI/component changes don't require tests; keep validating those via deploy-preview click-through. Run `npm test` (frontend unit tests), `node --test tests/worker-unit.test.mjs` (backend pure-function unit tests), and â€” for auth/permissions/admin-owner changes specifically â€” the relevant `tests/*.test.mjs` integration test (e.g. `node tests/auth.test.mjs`, `node tests/admin-owner.test.mjs`, `node tests/signup-recovery.test.mjs`, `node tests/admin-delete-user.test.mjs`, `node tests/self-delete-account.test.mjs`, `node tests/feedback.test.mjs`, `node tests/push-notifications.test.mjs`) before merging.

The full integration suite (`npm run test:integration`) also runs in CI via `.github/workflows/integration.yml`, but **only** on PRs that touch `worker/`, `tests/`, `shared/`, `migrations/`, or `wrangler.toml` (it's ~6 min â€” too slow for the every-push `ci.yml`). Each file gets its own isolated local D1 (`tests/_helpers.mjs`'s `--persist-to`) so IP-keyed rate-limit state can't bleed between files. Like the rest of CI it doesn't *gate* the deploy, but it's the pre-merge signal for these endpoints â€” don't merge an auth/permission change with it red.

**When the integration suite can't run in-session** (no `wrangler`/network, or a sandbox that can't boot Miniflare) and the CI job hasn't covered the change, verify an auth/permission change on the **deploy preview against throwaway accounts** before merging. Note the branch preview binds the **production** DB (see Databases â€” there's no preview DB), so those throwaway accounts are real prod rows: use obviously-fake credentials and delete them (via `DELETE /account` or a `d1_database_query`) when done. For a cross-tenant/permission check (cf. TODO #90): register two separate households via `/register` on the branch preview, confirm one can't reach the other's data or admin actions (expect `403`/`404`, not `200`), then remove both.

## Deployment

Pure-function logic has unit test coverage, run in CI on every push/PR, no D1/wrangler needed. Auth, permissions, and other DB-backed behavior have integration tests (`tests/*.test.mjs`) that spin up the real Worker locally against local D1 via `wrangler dev --local` (see `tests/_helpers.mjs`) â€” intentionally **not** wired into CI (too slow/flaky to run on every push); run on demand via `npm run test:integration` or `node tests/<file>.test.mjs`. UI/component behavior has no automated coverage and is validated by deploying:
- Worker changes: push to `main` â€” Cloudflare's Git integration runs `npx wrangler deploy` automatically.
- Frontend changes (`src/`): push to `main` â€” Cloudflare Pages runs `npm run build` and deploys `dist/` automatically. `npm install && npm run build` can also be run wherever Node is available (including inside a Claude Code session) to validate a change before pushing; `npm run dev` runs Vite's local dev server (starts at `/app.html`).
- Schema changes: add a new numbered file in `migrations/`, then see "Databases" and "Applying migrations" below â€” migrations are applied by hand to the production D1, and must be expand/contract.

### Rolling back a bad deploy

Every merge to `main` auto-deploys instantly and nothing gates it, so a bad deploy is live immediately â€” know the way back before you need it. There are three cases, by what shipped:

- **Code (frontend or Worker).** Revert the offending commit(s) on `main` (`git revert`, or a fresh PR that undoes them) and merge â€” the revert is just another push, so Cloudflare rebuilds and redeploys the previous behavior the same way. For an urgent stop-gap while preparing the revert, the Cloudflare dashboard can roll a Pages/Worker deployment back to a prior version directly (Deployments â†’ Rollback), independent of git.
- **A migration.** Migrations are applied by hand and are expand/contract, so a bad one is undone by a **new** reverse migration, not by editing history: write the compensating SQL (e.g. a follow-up `ADD COLUMN`'s data is left in place; a bad `CREATE TABLE`/`INDEX` gets a `DROP` in a later numbered file), apply it via `d1_database_query`, and record it in `d1_migrations` â€” the same runbook as applying one (below). Because expand/contract means the old code tolerated the new schema, the schema rarely needs urgent reversal; usually reverting the *code* is enough and the extra column/table just sits unused until a later cleanup migration drops it.
- **Both together.** Revert the code first (fastest way to restore behavior), then decide whether the schema change needs a reverse migration or can be left dormant.

Whichever path, if a rollback changes the built output (code path), it still follows the version rules â€” but a revert is usually paired with re-bumping/relanding, so use judgement rather than churning `VERSION` twice for a quick back-and-forth.

### Databases: one production D1 (previews share it)

There is a **single** D1 database (`database_id` in `wrangler.toml`), bound by the live Worker at shopping.mohibb.com **and** by every Cloudflare branch/commit *preview* deploy. There is deliberately **no** data isolation between prod and previews: `preview_database_id` only affects `wrangler dev`, not Workers Builds deploy previews, which always bind `database_id` â€” verified empirically (a login against the branch-preview Worker landed in the production DB). So **a preview reads and writes production data**: preview click-testing that creates accounts/items is mutating prod, and a preview runs its (new) code against the (current) prod schema.

Two facts drive the migration rules:
- Cloudflare's Git integration runs `wrangler deploy`/`versions upload` but **never** `d1 migrations apply` â€” migrations are always applied by hand, separately from the code deploy.
- On a merge to `main`, the migration lands slightly *before* the new code finishes building/deploying (~a minute), so for that window production runs the **old** code against the **new** schema â€” and a branch preview runs *new* code against a schema that may still be old until the migration is applied.

**Migrations must therefore be expand/contract â€” additive and backward-compatible with the currently-deployed code:**
- Only `ADD COLUMN` / `CREATE TABLE` / `CREATE INDEX`; never drop, rename, or retype anything live code reads.
- New columns are nullable or carry a `DEFAULT`, so existing rows and the old code's INSERTs stay valid.
- To remove or rename, do it over two releases: add-and-adopt first; drop the old column in a later migration only once no deployed code references it.

Because expand/contract guarantees the old code tolerates the new schema, the migration can be applied a little **ahead** of the merge so the branch preview can be click-tested against it â€” the additive change is invisible to the still-deployed prod code until the new code ships.

### Applying migrations without a local install

The developer installs nothing locally, and Claude Code sessions generally have **no** `CLOUDFLARE_API_TOKEN`/`wrangler login`, so `wrangler d1 migrations apply --remote` can't run from inside them. Instead, the allowed `mcp__Cloudflare_Developer_Platform__d1_database_query` tool (see `.claude/settings.json`) runs SQL against production D1 directly (Claude can do this in auto mode without prompting):

1. Write `migrations/00NN_*.sql` as usual (keeps the sequential-numbering CI check happy).
2. Apply it to production via `d1_database_query`, and record it: `INSERT INTO d1_migrations (name, applied_at) VALUES ('00NN_*.sql', datetime('now'));`. Because it's expand/contract, doing this before the merge is safe â€” the deployed prod code ignores the additive change, and the branch preview (same DB) can now be click-tested against the new schema.
3. Verify on the branch preview URL, then merge (Cloudflare auto-deploys the new code seconds later).
4. **Reconcile (run this every time, right after step 2's INSERT â€” the tracking table has no other guard).** `SELECT name FROM d1_migrations ORDER BY name` should be exactly the set of files in `migrations/`. Because the row is hand-inserted via MCP, nothing else verifies the two match: a missed INSERT, a filename typo, or a duplicate silently drifts the tracking table from the folder, and a future `wrangler d1 migrations apply` would then re-run or skip the wrong files. If they differ, fix the row (INSERT the missing one / correct the typo'd `name` / delete the dupe) before moving on â€” do not leave a known drift.
